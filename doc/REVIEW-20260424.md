## Chapter 3: The Imperative Programming Paradigm (`20-imperative.rst`)

**Description:** Lines 139–158 first show a `while`-loop version that mutates `var sum` and `var count`, then immediately show a `forEachRemaining` lambda version that reads `sum += s.length(); count += 1;` inside the lambda body. Java prohibits capturing mutable (non-effectively-final) local variables in a lambda; this code does not compile. **Proposed fix:** Fix both examples so they are self-consistent. For the `forEachRemaining` lambda, either (a) use an `int[]` wrapper trick (the idiomatic Java workaround), or (b) replace it with a `Stream.of(...).reduce(...)` pattern that avoids mutable capture. Remove the stray `});` from the while-loop example and ensure both snippets compile independently.

### Shortcoming 5: Monad and Monoid sections describe abstractions without a motivating example before the formal definition
**Description:** Lines 1055–1097 introduce Monoid and Monad with Wikipedia-style definitions ("a type with an associative binary operation and an identity element") before any motivating problem. Students who have not already encountered category theory will struggle to see why these abstractions are useful. The examples (integers, lists, strings) are given after the definition rather than before it, reversing the natural pedagogical order.  
**Proposed fix:** Invert the presentation: start with a concrete problem (e.g., "how do we generically combine any two log messages, or any two lists of errors, or any two sums?"), show that `foldLeft` requires knowing what the "zero" and "combine" operation are for each type, then introduce *Monoid* as the abstraction that captures exactly this pattern. Similarly for Monad: start with the `Option`-chaining problem already discussed earlier in the chapter, then name the pattern. This "motivate, then define, then generalise" structure is the standard pedagogical approach for abstract algebra in programming courses.

### Shortcoming 3: Semantic rules presented as opaque images, not typeset inference rules
**Description:** The structured operational semantics (SOS) sections use `.. image::` directives for evaluation.png, execution1.png, execution2.png, records1.png through records5.png to display inference rules. ## Chapter 7: The Concurrent Programming Paradigm (`60-concurrency.rst`)

### Shortcoming 1: Pedagogically inverted structure — technical content precedes motivation
**Description:** The "Core Elements" and "Examples Across Languages" sections (lines 7–116) present detailed technical content (threads, mutexes, actors, futures, a six-column comparison table) before the "Motivation" section (lines 232–255) that explains *why* concurrency is needed. The "Concurrency vs. Parallelism" section (lines 118–228) also precedes the motivation, compounding the problem.  
**Proposed fix:** Reorder the chapter: (1) Motivation — why concurrency matters; (2) Concurrency vs. Parallelism — key distinction; (3) Fine-grained vs. Coarse-grained — design dimension; (4) Core Elements — technical mechanisms; (5) Examples Across Languages — comparative table. ## Chapter 8: The Logic Programming Paradigm (`70-logic.rst`)

### Shortcoming 1: Chapter ends abruptly with the cut operator — no summary or broader context
**Description:** The chapter ends at line 213, immediately after the discussion of the cut operator (`!`). There is no summary section, no comparison of logic programming with other paradigms, no discussion of when to choose Prolog, and no mention of applications (natural language processing, theorem proving, AI planning, constraint satisfaction). The cut discussion itself ends with a mention of "Mercury, constraint logic programming" but neither is elaborated.  
**Proposed fix:** Add a "Summary and further reading" section that: (1) summarises the declarative, relational nature of logic programming in two to three sentences; (2) contrasts it with functional programming (both are declarative, but logic programming is *relational* — a predicate can be run in multiple directions); (3) lists two or three real-world applications; (4) provides three to five curated references (e.g., Sterling & Shapiro's *The Art of Prolog*, the SWI-Prolog manual, Bratko's *Prolog Programming for Artificial Intelligence*).

### Shortcoming 2: Claim that "Prolog does not have traditional I/O" is inaccurate
**Description:** Line 74 states "Prolog does not have traditional I/O" as justification for the `hello/0` fact approach. This is incorrect: SWI-Prolog has a full ISO-standard I/O library (`read/1`, `write/1`, `format/2`, file I/O, etc.). What Prolog lacks is a *main entry point* in the Java/Scala sense — it is query-driven, not procedure-driven. The inaccurate framing may confuse students.  
**Proposed fix:** Replace "Prolog does not have traditional I/O" with: "Prolog programs are query-driven rather than entry-point-driven: instead of a `main` method, you issue a query at the top-level. Prolog does support I/O predicates such as `write/1` and `nl/0`, which is how `hello/0` works above." This is accurate and pedagogically clearer.

### Shortcoming 3: The 8-queens example lacks a step-by-step explanation of the backtracking
**Description:** Lines 130–176 present the 8-queens Prolog solution, which involves `safe/1`, `no_attack/3`, and `queens/2`. While the code is correct, there is no prose explanation of how Prolog's backtracking explores the search tree, why `\+` (not-provable) is used instead of `\=`, or how the `member/2` predicate generates candidate values on backtracking. A student unfamiliar with Prolog will not understand *how* the solution works, only *that* it works.  
**Proposed fix:** Add a paragraph before the code that explains the generate-and-test strategy at a high level. After the code, add a `.. note::` block tracing the first few backtracking steps: "Prolog first tries `Q=1` for the first queen, then recursively places remaining queens; if `safe` fails at any point, Prolog backtracks and tries `Q=2`, and so on."
